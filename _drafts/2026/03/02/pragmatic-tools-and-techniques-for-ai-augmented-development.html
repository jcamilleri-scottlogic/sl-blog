--- title: Pragmatic tools and techniques for AI-Augmented development date: 2026-03-03 14:03:00 Z categories: - Artificial Intelligence tags: - AI - Artificial Intelligence - Agentic AI author: dkerr --- The landscape of AI-augmented development is changing rapidly. Over the past few months, I've leaned heavily into these tools to take open-source issues from "ready for development" straight through to mergeable solutions. One important note: as a software professional, I'm ultimately responsible for what reaches the codebase meaning full autonomous "vibe coded" pull-requests hitting the review queue is off-limits. Over time, I've started to internalise knowledge and experience of what is and isn't effective for me when performing AI augmented software development. My daily driver has been Github Copilot in Visual Studio Code, using models from the basic [GPT-5 Mini](https://developers.openai.com/api/docs/models/gpt-5-mini) (released August 2025) to the more premium [GPT-5.2 Codex](https://developers.openai.com/api/docs/models/gpt-5.2-codex) (released December 2025). I’ve documented these learnings to provide a point in time snapshot of what's worked for me so far. I've tried to order the tools and techniques roughly to mirror the typical lifecycle of a software issue. A fair warning: while my examples lean heavily on VS Code, I am confident the core strategies can be used whether you're using Claude Code, Cursor, Devin or the next tool. Writing up and sharing this also helps to better understand the processes myself and lowers the barrier to getting feedback from the wider community on how to wield all these tools more effectively. ## Establishing Project Baselines ![an image depicting the automated copilot instructions generation button in visual studio code](generate-copilot-instructions.png) Before diving into any issue, I've found it useful to establish project baselines via [generating copilot instructions](https://code.visualstudio.com/docs/copilot/customization/custom-instructions#_generate-custom-instructions-for-your-workspace) for your workspace. These help define coding standards, library preferences and architectural patterns before the AI attempts to make any adjustments to the codebase. These are included by default in every chat prompt, meaning it saves you the headache of including them yourself manually each time. Take the time to review and adapt these appropriately, don't rely on the AI to pick up everything. It can be particularly common to have unwritten rules and/or assumptions in a software project, this instructions file would be a great place to write any of those down. Revisit the instructions file as you work together with agents in the codebase. Agents frequently need fine tuning and guidance to avoid common pitfalls such as stalling indefinitely when choosing to run tests in watch mode. As you build up this instructions file you should start to see less and less teething issues when using agents. ## Faster Issue Contextualisation Getting up to speed on a new issue drains your daily cognitive bandwidth. Being able to shorten this isn't just saving time but also preserving your mental energy to use elsewhere. I've had great success using AI to summarise what can be dense, comment-heavy issues - stripping away non-essential or irrelevant conversation. Try prompting your agent to distil issues into core requirements, blockers and current consensus allowing yourself to focus quicker on the technical challenge at hand. Try not to limit yourself to textual based summarisations either - Modern multimodal models can digest almost anything attached to a ticket. Screenshots, stack traces, database schemas and even video recordings of a bug can all be fed to the agent to build a robust context before you write a single line of code. ## Analysis $\rightarrow$ Implementation $\rightarrow$ Reflection loop ![a diagram of the analysis implementation reflection loop](workflow-diagram.png) I've found that closing the feedback loop is especially important for AI augmented software development. We've seen that with the right constraints (e.g. a tightly bounded problem and comprehensive tests) AI can port large codebases with [relative ease](https://ikyle.me/blog/2025/swift-justhtml-porting-html5-parser-to-swift) and swiftness. Scaling this appproach down to individual issues is just as powerful. I've settled on an Analysis $\rightarrow$ Implementation $\rightarrow$ Reflection loop when working through an issue. Breaking the work into three distinct phases gives me the flexibility to interject whenever the AI starts to struggle or drift off course. ### Analysis Once _you've_ understood the issue - ideally using the faster contextualisation techniques mentioned earlier - it's time to build a lightweight harness to let the agent iterate in a feedback loop. I typically ask the agent to analyse the provided issue, and, adopting a TDD approach, write relevant failing tests that reproduce the bug or define the new feature. Be warned: the model may stumble even at this first hurdle, especially if the issue is difficult to reproduce (if at all). Take a moment here to reflect. Are the generated tests appropriate, relevant and offer good coverage? Are there any potential gaps or missing edge cases? Your goal herere is to verify that the AI agent fully understands the issue. At this point you should have a fairly decent idea in your head of a rough solution to the issue, giving you a solid point of comparison for the final reflection step. ### Implement With failing tests in place, the implementation phase should drive itself. I prompt the agent to implement a solution and use the test suite as the primary feedback mechanism (the classic Red/Green loop). The trick here is to instruct the AI to iterate until all tests pass. Example Prompt - I'm polite so include the word please : - ) > Now that we have failing tests, please implement the code required to resolve the issue. After your first attempt, automatically run the test suite. Use the test output as your primary feedback mechanism. If the tests fail, analyze the error, adjust your implementation, and run the tests again. Continue this Red/Green iteration loop until all tests pass. Crucially: Once all tests pass, stop immediately and await my review. Do not proceed to refactoring or further tasks. ### Reflection Arguably, this is the most important step of the loop, where I take on responsibility for the outlined solution. Because of this, I use reflective prompting to challenge the AI's outlined solution, particuarly if it deviates from how I assumed the issue would be solved. My typical lines of questioning include: #### Architectural Integrity * **What alternative solutions did you consider, why did you choose this one?** Outlines other potential pathways to a solution that may actually be preferrable * **What edge cases or unexpected inputs could cause this implementation to fail?** Gets the model to look for cracks in their own logic * **Are there any specific scenarios where this solution might introduce a race condition or state inconsistency?** Useful for any asynchronous or multi-threaded work #### Maintainability * **If a junior team member had to maintain this code in six months, what part would be the hardest for them to understand?** Neat trick to force AI to identify complex and/or unreadable logic that may warrant refactoring or commenting #### Security * **Are there any hidden performance bottlenecks or scaling issues in this approach?** Good catch-all for performance / inefficiency * **What potential security implications or vulnerabilities does this change introduce?** A mandatory sanity check, especially relevant if the solution handles user input or involves authentication Making the model defend and justify its logic is a great way to uncover edge cases or simpler paths that were glossed over in the first pass. I've also found use in switching to a completely different model at this point and ask it to run a blind code review on the newly implemented changes. ## Managing the context window Keeping an eye on your context window usage, particularly when you're deep in a long running chat thread with an agent can save you from diminishing performance. There are various levers you can pull to improve context efficiency. A major one to explore is the [Model Context Protocol](https://modelcontextprotocol.io/docs/getting-started/intro) (MCP), which lets your AI tools fetch specific, bite-sized context from your local environment on demand, saving you from pasting in whole files and burning through your token limits. Even with these tools, if context limits are a frequent problem, consider how you can break down larger pieces of work across multiple, isolated chats. More often than not, a bloated context window means you're simply trying to solve too much at once within the AI's current capabilities. ## Debugging AI Chat Agents have a big problem with explainability. Nobody can explain exactly what the models are doing at the root level, and because we are using bleeding-edge tools, the harness layer itself will frequently break in unexpected ways. However, you can give yourself a headstart by lifting the lid a little. Quite often, the misbehaviour isn't a deep problem it's just bad input at the harness level. I'd encourage you to have the confidence to dig down to that layer via [chat debug view](https://code.visualstudio.com/docs/copilot/chat/chat-debug-view) (equivalent tooling exists) which lets you see what context, prompts and tools were used when talking to an agent. This added transparency can help you course-correct effectively. ## Choosing the right context to include Finally, one area of frequent confusion that still isn't completely clear to me is knowing exactly when to opt for one type of AI context over another. The tooling landscape is pretty crowded and ever-changing. Frankly, the jury is still out on the true effectiveness of some of these granular methods. If you find yourself stuck wondering which context to use, I recommend relying on evergreen documentation provided by your tooling. For example, VS Code's documentation has a [quick reference table](https://code.visualstudio.com/docs/copilot/customization/overview#_quick-reference) that I frequently refer back to ad-hoc before reaching to include any additional AI context.
